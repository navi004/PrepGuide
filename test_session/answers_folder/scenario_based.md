## Technical Scenario-Based Questions

**1. You are tasked with designing the backend for a URL shortening service like bit.ly. Describe the high-level architecture. How would you generate unique short URLs, how would you store the mapping, and how would you design the system to handle millions of redirection requests per day with low latency?**
*Model Answer:*
*   **High-Level Architecture:** I'd use a load balancer distributing traffic to a fleet of stateless web servers. These servers would handle two main API endpoints: one for creating short URLs (`POST /shorten`) and one for handling redirections (`GET /{short_url}`).
*   **URL Generation:** To guarantee uniqueness and avoid collisions, I would use a central, auto-incrementing 64-bit integer from a database as a unique ID for each new URL. I would then Base62-encode this integer to create a short, clean alphanumeric string (e.g., `[a-z, A-Z, 0-9]`).
*   **Storage & Low Latency:** For storage, I'd use a dual-database approach. A primary relational database (like PostgreSQL) would store the permanent mapping and user data. For low-latency redirections, I would use a distributed cache like Redis. When a short URL is created, the mapping (`short_url -> long_url`) is written to both databases. Read requests for redirection would first check Redis; a cache hit results in a sub-millisecond response. A cache miss would query the primary DB and then populate the cache for future requests. This architecture can easily handle millions of requests.

**2. Imagine you are designing a feature for Google Photos that automatically suggests tags for an uploaded image (e.g., "beach," "dog," "sunset"). Outline the system design for this feature, from the moment a user uploads a photo to when they see the suggested tags.**
*Model Answer:*
1.  **Upload & Trigger:** The user uploads a photo to a distributed object store like Google Cloud Storage. This upload event publishes a message to a message queue like Google Pub/Sub, containing the image's location and user ID.
2.  **Asynchronous Processing:** A fleet of worker services, running on a scalable platform like Kubernetes or Cloud Run, subscribes to this queue. A worker picks up a message, downloads the image, and possibly creates thumbnails.
3.  **ML Inference:** The worker sends the image to a highly optimized, pre-trained computer vision model (likely a Vision Transformer or a large CNN) hosted on an ML serving platform like Vertex AI. This service returns a list of potential tags and their confidence scores (e.g., `{"beach": 0.95, "sky": 0.92, "water": 0.88}`).
4.  **Storage & Delivery:** The worker service processes these results, filters by a confidence threshold, and stores the final tags in a NoSQL database (like Firestore) linked to the image's metadata. Finally, a notification is sent to the user's client app via a push service, or the client polls for the updated metadata, causing the suggested tags to appear in the UI.

**3. You need to build a system that can process a massive stream of real-time event data (e.g., clicks on a website) and provide a dashboard showing the top 10 most frequent event types over the last 5 minutes. How would you approach this problem? Describe the components of your system.**
*Model Answer:*
1.  **Data Ingestion:** I'd use a high-throughput message broker like Apache Kafka to ingest the raw clickstream data. This decouples the data producers from the consumers and handles backpressure.
2.  **Stream Processing:** I'd use a stream processing framework like Apache Flink or Spark Streaming. The application would consume events from Kafka and process them in a 5-minute sliding window.
3.  **Top-K Algorithm:** To efficiently find the top 10 events within the stream without storing all events, I'd use a probabilistic approach. The "Count-Min Sketch" algorithm can estimate frequencies of events in a memory-efficient way, combined with a Min-Heap of size 10 to keep track of the current top 10 candidates.
4.  **Serving Layer & Dashboard:** Every few seconds, the stream processor would write the latest top-10 list to a fast, in-memory data store like Redis. A simple backend API would query Redis and expose this data to a frontend dashboard, which would poll the API to display the results in near real-time.

**4. You are building a collaborative text editor similar to Google Docs. Describe the data structure you would use to represent the document on the server and the algorithm or protocol you would implement to handle concurrent edits from multiple users while ensuring consistency for everyone.**
*Model Answer:*
*   **Data Structure:** Instead of a simple string or array of characters which would be inefficient for frequent insertions and deletions, I would use a more advanced data structure like a **Rope** or a **Piece Table**. Both are excellent for representing a document's state, as they handle edits by manipulating pointers or references rather than copying large blocks of memory, making them very fast.
*   **Concurrency Algorithm:** I would implement **Operational Transformation (OT)**. In an OT-based system, each user's edit (an "operation") is sent to a central server. The server maintains the authoritative state of the document. When it receives an operation, it transforms it against any other operations that have occurred concurrently, ensuring the user's intent is preserved in the correct context. It then broadcasts the transformed operation to all other clients. This client-server model with OT is a proven, robust way to ensure all users converge on an identical document state.

**5. Design a rate-limiting system for a public API. The system should be able to enforce limits like "100 requests per user per minute." Explain how you would track request counts for each user efficiently and how the system would behave in a distributed environment with multiple API servers.**
*Model Answer:*
*   **Algorithm & Tracking:** I would use the **Fixed Window Counter** algorithm implemented in a centralized, in-memory data store like **Redis** for efficiency and scalability. For each user, I would create a key, for example `ratelimit:{user_id}:{timestamp}` where the timestamp is truncated to the current minute. When a request comes in, the system would use the `INCR` command on this key. If the count is 1, it would also set an `EXPIRE` time of 60 seconds on the key.
*   **Distributed Environment:** Using a centralized Redis instance is key for a distributed environment. An API Gateway or a middleware layer on each API server would be responsible for the rate-limiting logic. Before processing a request, it would call Redis with the user's key.
    1.  It increments the counter.
    2.  If the returned count is below the limit (e.g., <= 100), the request is allowed to proceed to the backend service.
    3.  If the count exceeds the limit, the gateway immediately rejects the request with a `429 Too Many Requests` HTTP status code, protecting the backend services from being overwhelmed.
